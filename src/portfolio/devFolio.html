battleship

view code for manual play view code for AI play

In one of my fundamental computer science courses at Northeastern, Object Oriented Design, we were assigned to develop a functioning game of Battleship, played through the console of an IDE. The goal of this assignment was to understand the design concept of MVC, or model, view, and control. This was built entirely with Java, and includes extensive use of inheritance, arrays, and maps.

The project comes in two parts. The first part was to develop a Battleship program that allows a human player to battle against a computerized player, that takes shots according to the code, whether it be random or strategic. Click "view code for manual play" to see this code. The second part of this assignment was to take our code from part one, make it connect to a server, and have other AI players play against the computerized player from our code. On the final day of this assignment, students were asked to set up their programs to have their codes battle each other, tournament style. Conceptually, this meant connecting the opponent's AI and having it act as the human player, providing input for our computerized player to respond to and play against.

Part two of this project required us to rely on JSON objects and messages that align with the messages sent by the server. This was done through a proxy, which delivered and took in JSON arguments and messages between our AI and the opponent AI's plays sent by the server. Additionally, part two was completed in two-person teams and my team ended up using my partner's code, which is why the code for AI play appears different from the code for manual play. I worked extensively on the developing the proxy to match server input and outputs, creating various JSON objects, and writing code to serialize and deserialize JsonNode arguments and responses.

bullet journal

view code

Creating a functioning, visually cohesive, and assistive bullet journal was the last project for Object Oriented Design. The key functions for the program were event and task creation that would be shown on a week view, saving the bullet journal as files, and setting maximum numbers of events and tasks. From there, we were tasked to gradually add features to the program, including preset and custom themes, queue of tasks, progress bars of task completion, privacy lock, and a splash screen. For this project, we worked in teams of three.

Within my group, I was the only student with a design background, so I took it upon myself to design the UI of the program through SceneBuilder. This required thorough understanding of the JavaFx hierarchy, as the program's ability to progress relied on proper use of flexible VBox's. Additionally, I created buttons with the ability to change the colors of the program to either a preset theme or a theme of the user's choice using a color picker. I also worked on setting up the task queue and progress bars, ensuring that checking off one box under a weekday would both check off the box in the task queue as well as further the weekday's progress bar. Once this was completed, I worked on the addition of a splash screen that featured a privacy lock. The splash screen was created with fading transitions of a VBox that contained text for the privacy lock. The privacy lock on the screen is dependent on the name of the user's system username. That allows for users to have different passcodes across different devices.

This bullet journal project was the project that I really had fun with. I had so much fun exploring the internet for ways to incorporate my creative ideas into my code. A lot of the features that were implemented were not taught in class, and it was incredibly invigorating seeing how creative I could get with my program.
